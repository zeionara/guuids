# Installing Gentoo

This guide covers `Gentoo` setup on the `Orange Pi 4 Pro`. In a [dedicated guide][opi-6plus-windows] I've described `multiboot` configuration on `Orange Pi 6 Plus`, which involved installing `Windows`, `Gentoo` and `Ubuntu` on the same device. However, it appears that `Orange Pi 4 Pro` is different - it relies on `U-boot` and doesn't support `UEFI`, that is why its' capabilities are more limited. Nevertheless, it is still possible to configure `Gentoo` here, and on this page I provide a detailed explanation how to do that.

## Booting the device

First of all, since `Orange Pi 4 Pro` isn't shipped with `BIOS` or `UEFI`, you need a good `SD` card with at least `8 Gb` capacity for initial bootloader configuration, I recommend using `SD` cards from `Netac`. Also you need an `NVMe` drive which will be the long-term storage device for `Orange Pi 4 Pro`. After the storage devices are ready, you need to download `Ubuntu` image provided by the board vendor following [this link][opi-4pro-ubuntu].  

Unzip the downloaded image and verify the file integrity:  

```sh
cd ~/Downloads
7zz x Orangepi4pro_1.0.6_ubuntu_jammy_server_linux5.15.147.7z
sha256sum -c Orangepi4pro_1.0.6_ubuntu_jammy_server_linux5.15.147.img.sha
```

Then you need to burn the `Ubuntu` image to both - `SD` card and `NVMe` device using these commands, where `/dev/mmcblk1` corresponds to the `SD` card and `/dev/sda` corresponds to the `NVMe` drive:  

```sh
sudo dd if=Orangepi4pro_1.0.6_ubuntu_jammy_server_linux5.15.147.img.sha of=/dev/mmcblk1 bs=4M status=progress conv=fsync
sudo dd if=Orangepi4pro_1.0.6_ubuntu_jammy_server_linux5.15.147.img.sha of=/dev/sda bs=4M status=progress conv=fsync
```

Insert prepared `NVMe` drive and `SD` card into the board, connect the board to a display through `HDMI` interface and to the Internet via `Ethernet` cable, and then connect to a power source, the device will boot automatically without pressing any additional buttons. After that you'll see the `Orange Pi` logo with message from bootloader. In a few moments `Ubuntu` will start up and now you're ready to configure the bootloader.

## Configuring the on-board bootloader

By default the device doesn't come with pre-installed bootloader, but it has `SPI Flash`, which you can use to store the bootloader on the board itself. Before doing that, reset `SD` card `UUID` to avoid collisions with `NVMe` partition using command like this, where `/dev/mmcblk1p1` corresponds to the `Ubuntu` partition on the `SD` card:

```sh
sudo tune2fs -U random /dev/mmcblk1p1
```

Optionally, verify that `UUIDs` are now different for `Ubuntu` partitions on `SD` card and `NVMe` device:

```sh
blkid
```

::: danger Do not skip the UUID update step

It is very important to make sure that all partitions available to the system have different `UUIDs`, so before proceeding make sure that this assumption is true.

:::

Make sure that `/mnt/nvme_opi_root/boot/orangepiEnv.txt` points to the correct partition (which is `Ubuntu` partition on the `NVMe` device), where `/mnt/nvme_opi_root` is the mount point to the `Ubuntu` partiton on the `NVMe` device:

```sh
cat /mnt/nvme_opi_root/boot/orangepiEnv.txt
```

After eliminating collisions between `UUIDs` and making sure that `orangepiEnv` points to the correct partition, run the following command from the `Ubuntu` terminal for installing bootloader into on-board `SPI Flash`:

```sh
sudo orangepi-config
```

Then navigate to `System â†’ Install â†’ Install/Update bootloader on SPI Flash` in the menu which appears. After the installation completes, create a separate partition on the `NVMe` device for installing `Gentoo` using [this guide][gentoo-partitioning].


::: tip Creating cache partition

I recommend creating a separate `cache` partition, which can be used for storing large volumes of data for `docker`, `pip`, `huggingface`, etc, so that the resulting `/etc/fstab` file with partition layout in `Gentoo` will look like this:

```sh
# /etc/fstab: static file system information.
#
# See the manpage fstab(5) for more information.
#
# NOTE: The root filesystem should have a pass number of either 0 or 1.
#       All other filesystems should have a pass number of 0 or greater than 1.
#
# NOTE: Even though we list ext4 as the type here, it will work with ext2/ext3
#       filesystems.  This just tells the kernel to use the ext4 driver.
#
# NOTE: You can use full paths to devices like /dev/sda3, but it is often
#       more reliable to use filesystem labels or UUIDs. See your filesystem
#       documentation for details on setting a label. To obtain the UUID, use
#       the blkid(8) command.

# <fs>                  <mountpoint>    <type>          <opts>          <dump> <pass>

#LABEL=boot             /boot           ext4            defaults        1 2
#UUID=58e72203-57d1-4497-81ad-97655bd56494              /               xfs             defaults                0 1
#LABEL=swap             none            swap            sw              0 0
#/dev/cdrom             /mnt/cdrom      auto            noauto,ro       0 0
UUID=<gentoo-partition>               /       ext4            defaults,noatime        0 1
UUID=<ubuntu-partition>               /mnt/opi_root   ext4            defaults,noatime        0 0
UUID=<cache-partition>               /mnt/cache      ext4            defaults,noatime        0 0
```

:::


After finishing the partition layout configuration disconnect the `SD` card and reboot - the `Ubuntu` system from `NVMe` should be brought up:

```sh
sudo reboot
```

## Configuring Gentoo

Now that you've got a self-sufficient `Ubuntu` environment you can start installing `Gentoo` on a separate partition. If anything goes wrong in the process, you can just insert `SD` card with valid `Ubuntu` partition - the linux kernel will be loaded from the `SD` card, but `Ubuntu` partition on `NVMe` device will be used as a `rootfs`. From there you will be able to start inspecting the problem and fixing `Gentoo` installation issues.  

On the previous step we've created a separate partition for installing gentoo using [this guide][gentoo-partitioning]. Suppose that `Ubuntu` partition is called `/dev/nvme0n1p1` and `Gentoo` partition is called `/dev/nvme0n1p2`. Then your next step is mounting `gentoo` into the currently running system:

```sh
sudo mkdir /mnt/gentoo
sudo mount /dev/nvme0n1p2 /mnt/gentoo
```

Download `stage3` archive to the `Gentoo` root (replace the mentioned `URL` with the latest one which can be found on [gentoo downloads page][gentoo-downloads]):

```sh
wget https://distfiles.gentoo.org/releases/arm64/autobuilds/20260517T234559Z/stage3-arm64-openrc-20260517T234559Z.tar.xz -L /mnt/gentoo
```

Unzip the downloaded archive:  

```sh
sudo tar xpvf /mnt/gentoo/stage3-*.tar.xz --xattrs-include='*.*' --numeric-owner -C /mnt/gentoo
```

Edit `make.conf` according to [these instructions][gentoo-make-conf-init]:

```sh
sudo vim /mnt/gentoo/etc/portage/make.conf
```

Copy `DNS` information to the `Gentoo` partition:

```sh
sudo cp --dereference /etc/resolv.conf /mnt/gentoo/etc
```

Then you can `chroot` into `Gentoo` environment using the following sequence of commands (I recommend running these commands in a `tmux` session for being able to detach from the session without losing context on long-running tasks):

```sh
sudo mount --types proc /proc /mnt/gentoo/proc
sudo mount --rbind /sys /mnt/gentoo/sys
sudo mount --make-rslave /mnt/gentoo/sys
sudo mount --rbind /dev /mnt/gentoo/dev
sudo mount --make-rslave /mnt/gentoo/dev
sudo mount --bind /run /mnt/gentoo/run
sudo mount --make-slave /mnt/gentoo/run
test -L /dev/shm && sudo rm /dev/shm && sudo mkdir /dev/shm
sudo mount --types tmpfs --options nosuid,nodev,noexec shm /dev/shm
sudo chmod 1777 /dev/shm /run/shm
sudo chroot /mnt/gentoo /bin/bash
> export PS1="(chroot) ${PS1}"
```

Configure `locale` following [these steps][gentoo-locale]. Then proceed from [there][gentoo-base], starting at `emerge-webrsync` execution and ending at wireless network tools installation (set `/etc/portage/package.use/00video-cards` to `*/* VIDEO_CARDS: -* panfrost`).

::: tip Skipping linux kernel compilation

In this guide we will be using vendor kernel, which is shipped with `Ubuntu` partition - that is why there is no need to compile linux kernel while installing `Gentoo`.

:::


Before exiting chroot, there is one more convenience step that could be done for being able to access the system remotely. To perform this step you need to install `openssh` package:

```sh
sudo emerge --ask net-misc/openssh
```

Then add `sshd` service to the `default` runlevel:

```sh
rc-update add sshd default
```

And optionally allow `root` login by editing `/etc/ssh/sshd_config`:

```sh
# Authentication:

#LoginGraceTime 2m
#PermitRootLogin prohibit-password
PermitRootLogin yes  # [!code ++]
```

Then exit chroot by pressing `Ctrl+D`.

## Installing missing software

The `Gentoo` partition prepared in the previous section will be used as a `rootfs`, and it should contain `kernel` modules for enabling the board hardware. To make the modules available to `Gentoo` instance, we need to copy the kernel modules form `Ubuntu` to `Gentoo` partition. First of all, check which kernel is currently loaded:

```sh
uname -r
```

This command outputs something like:

```sh
5.15.147-sun60iw2
```

Then copy modules from the folder which corresponds to this kernel on `Ubuntu` partition:

```sh
sudo mkdir -p /mnt/gentoo/lib/modules
sudo cp -a /lib/modules/5.15.147-sun60iw2 /mnt/gentoo/lib/modules/
```

Copy `wifi` firmware to `Gentoo` partition using this command:

```sh
sudo cp -a /lib/firmware/aic* /mnt/gentoo/lib/firmware/
```

And configure the `wifi` kernel module to load on startup by default:

```sh
sudo mkdir -p /mnt/gentoo/etc/modules-load.d
echo 'aic8800_fdrv' | sudo tee /etc/modules-load.d/wifi.conf >/dev/null
```

## Switching from Ubuntu to Gentoo

Finally, we need to change the system which will be started by bootloader. This is achieved by modifying the root parition in the file `/boot/orangepiEnv.txt`:

```sh
verbosity=1
bootlogo=false # [!code --]
bootlogo=true # [!code ++]
overlay_prefix=sun60i-a733
fdtfile=allwinner/sun60i-a733-orangepi-4-pro.dtb
rootdev=UUID=<ubuntu-uuid> # [!code --]
rootdev=UUID=<gentoo-uuid> # [!code ++]
rootfstype=ext4
```

::: warning Changing bootlogo

Notice that `bootlogo` flag has to be changed from `false` to `true` - this is important because it impacts the boot sequence. If the flag is not updated, `Gentoo` will fail to start up.

:::

Then reboot the device to get into the newly installed `Gentoo` system:

```sh
sudo reboot
```
