# Multiboot Setup

This guide covers multiboot setup configuration. As a prerequisite, you need to install `Gentoo` first on your own or following [the corresponding guide](/guuids/Gentoo/1.%20Installing%20the%20base%20system). Also make sure you've enough space for creating multiple partitions which contain the operating systems.

## Dual boot (Gentoo + Ubuntu 24.04 Desktop Orange Pi 6 Plus Edition)

After you've installed `Gentoo` on `Orange Pi 6 Plus`, download the [customized ubuntu image](https://drive.google.com/drive/folders/14vrKI_sFjvkravyDRIKskRYVMDfJB2kK) provided by the board developers, and put it at `/home/$USER/Downloads/Orangepi6plus_1.0.2_ubuntu_noble_desktop_gnome_linux6.6.89.img.xz`. Also download the `sha` signature from the same folder and put it at `/home/$USER/Downloads/Orangepi6plus_1.0.2_ubuntu_noble_desktop_gnome_linux6.6.89.img.xz.sha`.

Verify package integrity:

```sh
cd /home/$USER/Downloads
sha256sum -c Orangepi6plus_1.0.2_ubuntu_noble_desktop_gnome_linux6.6.89.img.xz.sha
```

If it says `FAILED`, download the package again, if it says `OK`, extract the image:

```sh
unxz Orangepi6plus_1.0.2_ubuntu_noble_desktop_gnome_linux6.6.89.img.xz
```

Associate the disk image with the first available `loop` device:

```sh
sudo losetup -fP Orangepi6plus_1.0.2_ubuntu_noble_desktop_gnome_linux6.6.89.img
```

List available partitions to verify that the operation completed successfully:

```sh
lsblk /dev/loop0
```

This command should display two partitions - the first is `efi` partition and the second is the root partition of ubuntu system. Mount the second partition:

```sh
sudo mkdir /mnt/img
sudo mount /dev/loop0p2 /mnt/img
```

Then mount the partition on disk, to which you want to install ubuntu for dual boot, for examble (replace `/dev/nvme0n1p3` with your partition):

```sh
sudo mkdir /mnt/ubu
sudo mount /dev/nvme0n1p3 /mnt/ubu
```

Make sure that the target partition is empty:

```sh
ls -alh /mnt/ubu
```

::: tip Formatting the target partition

If the target partition is not empty, then unmount it:

```sh
sudo umount /dev/nvme0n1p3
```

And recreate the filesystem (this operation will erase all data on the partition):

```sh
sudo mkfs.ext4 /dev/nvme0n1p3
```

Then mount again:

```sh
sudo mount /dev/nvme0n1p3 /mnt/ubu
```

:::

Run the following command to copy all files from the second partition of the ubuntu disk image to target partition (pay attention to the trailing slashes - they are essential):

```sh
sudo rsync -aAXv --exclude={"/mnt/target/*","/proc/*","/sys/*","/dev/*","/tmp/*"} /mnt/img/ /mnt/ubu/
```

Identify the `uuid` and `partuuid` of the target partition:

```sh
sudo blkid
```

Update `/mnt/ubu/etc/fstab` to match the new parition like this:

```sh
# /etc/fstab: static file system information.
#
# Use 'blkid' to print the universally unique identifier for a device; this may
# be used with UUID= as a more robust way to name devices that works even if
# disks are added and removed. See fstab(5).
#
# <file system>                           <mount point>   <type>  <options>       <dump>  <pass>
UUID=3f2cc97f-05e4-42ca-8dc2-2fc0a80098b0               /       ext4            defaults,noatime        0 1  # [!code ++]
```

Then make backup of the `grub` config:

```sh
sudo cp /boot/grub/grub.cfg /boot/grub/grub.cfg.old
```

Since `os-prober` is not available on `arm64`, you need to insert the `menuentry` for `ubuntu` manually, pay attention to the `uuid` (`3f2cc97f-05e4-42ca-8dc2-2fc0a80098b0` in our case) and `partuuid` (`9a7cb3ce-0e73-4c38-baac-6cbef0ff9639` in our case) which should point to the correct partition:

```sh
menuentry 'Ubuntu 24.04 Desktop Orange Pi 6 Plus Edition' --class ubuntu --class gnu-linux --class gnu --class os {
    search --no-floppy --fs-uuid --set=root 3f2cc97f-05e4-42ca-8dc2-2fc0a80098b0
    linux /boot/Image \
        console=ttyAMA2,115200 \
        efi=noruntime \
        earlycon=pl011,0x040d0000 \
        arm-smmu-v3.disable_bypass=0 \
        cma=640M \
        acpi=force splash \
        loglevel=4 \
        pcie_aspm=off \
        root=PARTUUID="9a7cb3ce-0e73-4c38-baac-6cbef0ff9639" rootwait rw
}
```

Then detach the mounted partitions:

```sh
sudo umount /dev/loop0p2
sudo umount /dev/nvme0n1p3
```

Detach the `loop` device from the disk image:

```sh
sudo losetup -d /dev/loop0
```

And try rebooting:

```sh
sudo reboot
```

Now you shold see `ubuntu` menu entry in the grub menu, the system should launch seamlessly.
